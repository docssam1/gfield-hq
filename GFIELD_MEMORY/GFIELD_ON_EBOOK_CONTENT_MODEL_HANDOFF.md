## 0. 한 줄 결론

GFIELD-ON은 단순 PDF 자료실이나 AI 플랫폼이 아니다.  
**지필드가 오래전부터 고민해 온 “어떻게 가르칠까 / 성공하는 공부 습관을 어떻게 만들까”의 결과물을 담는, 승인된 학생 전용 시크릿 학습 서재**다.

1차 핵심 제품은 다음이다.

```text
영상과 교재가 같이 보이는 지필드 eBook 학습 뷰어
+
시크릿 서재 감성의 홈페이지/플랫폼 프리뷰
```

AI-DOC / AI-DOT은 메인 기능이 아니라, 작은 보조 기능이다.

## 1. 브랜드·홈페이지 방향

### 1.1 핵심 서사

지필드는 기술을 먼저 고민한 것이 아니다.

```text
어떻게 하면 아이를 제대로 가르칠 수 있을까?
어떻게 하면 아이가 스스로 성공하는 공부 습관을 만들 수 있을까?
```

전자펜, 맞춤 영상, 첨삭, eBook, GFIELD-ON은 모두 그 고민에서 나온 도구다.

### 1.2 메인 카피 후보

```text
어느 누구보다 앞서 준비해 온
지필드의 비밀 서재가 열립니다.
```

```text
성공하는 공부 습관의 비밀,
지필드의 10년이 GFIELD-ON으로 열립니다.
```

```text
승인된 학생만을 위한
지필드의 시크릿 학습 공간.
```

### 1.3 14년 전 기록의 의미

14년 전 전자펜 첨삭·맞춤 영상 기록은 “디지털 학습을 오래전부터 고민했다”는 기술 자랑이 아니다.

정확한 의미:

```text
지필드는 오래전부터
학생이 직접 푼 풀이를 확인하고,
한 문제 한 문제를 살피고,
이해가 안 되는 문제는 한 명의 학생을 위해 다시 설명하는 방식을 고민했다.
```

GFIELD-ON은 그 철학이 쌓여 생긴 비밀 서재다.

### 1.4 오프닝 연출

소개 화면은 기능 설명보다 회고록처럼 지나가는 히스토리 릴이 좋다.

```text
14년 전 전자펜 영상
→ 네이버 카페 기록
→ 맞춤 영상 첨삭
→ 한 문제 한 문제의 기록
→ 교재 제작
→ 수업 자료
→ 학부모 반응
→ 지금의 GFIELD-ON
```

연출 방향:

```text
옛 영상 썸네일 0.5초
카페 글 캡처 0.5초
전자펜 필기 장면 0.5초
교재 표지들 0.5초
학부모 카톡 반응 0.5초
플랫폼 화면 1초
마지막 GFIELD-ON 로고 고정
```

### 1.5 외부생 유입 전략

겉으로는 재원생·승인 학생 전용 시크릿 공간이다.  
하지만 마케팅 목적은 외부 학부모가 이렇게 느끼게 하는 것이다.

```text
지필드에 들어오면 이런 관리와 자료를 받는구나.
우리 아이도 저 안에 들어가게 하고 싶다.
```

따라서 홈페이지는 노골적인 모집몰이 아니라 **공개하지 않는 공간을 살짝 보여주는 프리뷰**처럼 보여야 한다.

### 1.6 홈페이지 화면 역할 분리

```text
1. 소개 화면
- 지필드 10년의 역사
- 성공하는 공부 습관의 비밀
- 승인된 학생만의 시크릿 서재
- 링크 덕지덕지 금지

2. 로그인 이후 플랫폼 프리뷰
- 자료실/업로드 이미지 기반의 실제 플랫폼 목업 중심
- 수학·영어·과학 확장감 표시
- 이미지가 주인공, 텍스트는 최소화

3. AI-DOT 표현
- 플랫폼 위 작은 캐릭터/이미지 + 말풍선 한 줄
- 기능 섹션처럼 크게 만들지 않음

4. 학부모 반응
- 카톡 캡처 느낌은 강력함
- 실제 이름/프로필/개인정보는 반드시 흐림 처리
```

AI-DOC 링크를 소개 페이지에 덕지덕지 달지 않는다.  
GitHub 게시판도 사용자 화면에 노출하지 않는다. 내부 자료 원천/관리자 참고용이다.

## 2. viewer-v2 핵심 정체성

### 2.1 viewer-v2는 AI 플랫폼이 아니다

정체성:

```text
영상과 교재가 같이 보이는 지필드 eBook 학습 뷰어
```

기본 화면:

```text
상단
- 강의 버튼 배열
- 책만 보기
- 서재로

왼쪽 40%
- 선택된 영상 iframe
- 영상만 보기
- 새창으로 열기
- 작은 AI-DOT 말풍선 정도

오른쪽 60%
- 교재 eBook 페이지
- 한 페이지가 영역 너비에 딱 맞음
- 좌우 얇은 화살표
- 휠로 페이지 이동
```

기본 학습 모드 비율:

```text
영상 : 교재 = 4 : 6
```

### 2.2 교재 표시 방식

PDF iframe 주력은 피한다.  
디딤돌 eBook처럼 페이지 이미지 기반이 안정적이다.

예시:

```text
books/hs-thinking-basic/assets/pages/1.jpg
books/hs-thinking-basic/assets/pages/2.jpg
...
```

또는:

```text
books/hs-thinking-basic/assets/pages/page-001.webp
books/hs-thinking-basic/assets/pages/page-002.webp
...
```

교재 원본은 PDF/HTML로 보관 가능하지만, 뷰어 화면은 다음 구조가 좋다.

```text
페이지 이미지 = 책 화면
HTML/JS = 영상, 버튼, 페이지 이동, 책만 보기, 인쇄, 워터마크를 조종하는 엔진
```

### 2.3 학습 모드

필수:

```text
- 영상 4 : 교재 6
- 교재 한 페이지가 오른쪽 영역에 딱 맞음
- 아래 가로 스크롤바 없음
- 좌우 얇은 화살표
- 마우스 휠 위 → 이전 페이지
- 마우스 휠 아래 → 다음 페이지
- 상단 강의 버튼 클릭 → 영상 전환 + 교재 페이지 이동
```

불필요:

```text
- 1.4배 / 1.7배 같은 임의 확대 배율 기본값
- 문제 하나만 클릭해서 확대
- 문제 단위 crop 확대
- 아래 스크롤바로 책 보기
```

확대는 문제 단위가 아니라 페이지 단위다.

### 2.4 상단 강의 버튼

상단 버튼은 admin 데이터에서 자동 생성한다.

동작:

```text
상단 버튼 클릭
→ 왼쪽 영상 iframe 전환
→ 오른쪽 교재 page 이동
→ 선택 버튼 표시
```

상단 버튼은 가로 스크롤 기본 구조로 만들지 않는다.

권장:

```text
1~6개: 한 줄
7~12개: 두 줄
13개 이상: 더보기
```

### 2.5 책만 보기 / eBook 모드

책만 보기 클릭 시:

```text
영상 영역 숨김
교재 100% 확장
```

필수 기능:

```text
- 한 페이지 보기
- 두 쪽 펼침 보기
- 좌우 얇은 화살표
- 휠로 이전/다음 페이지 넘김
- 오른쪽 확대/축소 컨트롤
- 전체화면
- 현재 쪽수 유지
- #page=현재쪽 구조 고려
```

두 쪽 보기에서는 가운데 접힌 부분 음영을 넣으면 책 느낌이 살아난다.

디딤돌 CSS 참고 핵심:

```text
.page_center_gradient
#page_prev_btn
#page_next_btn
#control 확대/축소 바
#pageframe transition 0.3~0.4초
```

과한 3D 책장 넘김 효과는 필요 없다.  
0.3~0.4초 정도의 부드러운 전환이면 충분하다.

### 2.6 인쇄 모드

인쇄는 새 주소로 빠지는 것이 아니라, 같은 viewer 주소 안에서 인쇄 상태로 전환한다.

```text
viewer-v2.html#page=2
→ 인쇄 버튼
→ 같은 뷰어 내부에서 인쇄 미리보기 모드
```

기본값:

```text
현재 페이지 한 장 인쇄
```

선택:

```text
- 현재 페이지
- 범위 지정
- 전체 페이지
- 일반화질
- 고화질
```

고화질/전체 페이지 인쇄는 경고가 필요하다.

```text
고화질 또는 전체 페이지 인쇄는 PC 사양에 따라 일부 페이지가 누락될 수 있습니다.
권장 방식은 현재 페이지 또는 짧은 범위 인쇄입니다.
```

워터마크는 인쇄 모드에서만 적용한다.

예:

```text
GFIELD-ON · 승인 학생 전용 · 학생명
```

### 3.2 페이지 이미지 + 컴포넌트 레이어

디딤돌 구조는 단순 이미지 뷰어가 아니라, 페이지 이미지 위에 컴포넌트를 얹을 수 있는 구조다.

가능한 컴포넌트:

```text
text
link
image
imageSlider
audio
video
youtube
```

우리 1차 구현에서는 페이지 위 컴포넌트를 본격 구현하지 않는다.  
하지만 구조를 닫아버리면 안 된다.

권장 구조:

```json
{
  "page": 37,
  "src": "assets/pages/37.jpg",
  "components": []
}
```

나중에 확장:

```json
{
  "page": 37,
  "src": "assets/pages/37.jpg",
  "components": [
    {
      "type": "imageSlider",
      "id": "slider_37_1",
      "x": 420,
      "y": 180,
      "width": 300,
      "images": [
        { "src": "contents/sample1.jpg", "link": "" },
        { "src": "contents/sample2.jpg", "link": "" }
      ]
    }
  ]
}
```

정리:

```text
페이지 이미지는 책의 바닥.
컴포넌트는 그 위에 얹는 레이어.
영상 학습은 우선 왼쪽 패널.
페이지 위 컴포넌트는 2차 확장.
```

## 4. 검색·드래그 질문·텍스트 레이어

### 4.1 HTML을 고집했던 이유

HTML을 고집한 이유는 교재를 보기 위해서가 아니라, 교재를 조종하기 위해서다.

```text
영상 버튼 클릭
→ 영상 변경
→ 교재 페이지 이동
→ 책만 보기
→ 인쇄
→ 워터마크
→ 검색
→ 드래그 질문
→ 유사문제 템플릿 연결
```

PDF iframe만으로는 이 흐름이 답답하다.

### 4.2 검색 기능

디딤돌 검색은 페이지 이미지 뒤에 검색용 텍스트 데이터가 있기 때문에 가능하다.

우리 구조:

```text
page-004.webp
= 실제 교재 이미지

page-004-text.json
= OCR 또는 원본 텍스트 추출 결과 + 위치 좌표

search-index.json
= 전체 페이지 검색용 인덱스
```

예:

```json
{
  "page": 4,
  "blockId": "p004_07",
  "text": "삼각형의 성질을 이용하여 구하기",
  "bbox": [120, 340, 620, 375]
}
```

검색 동작:

```text
검색어: 삼각형
→ 결과 목록 표시
→ 썸네일 표시
→ 검색어 빨간색 강조
→ 클릭하면 해당 페이지 이동
→ 가능하면 해당 위치 하이라이트
```

### 4.3 PaddleOCR / 텍스트 추출

PaddleOCR은 필수 이름이 아니라 후보 도구다.

가능한 방식:

```text
1. 원본 HTML/PDF에 텍스트가 살아 있음
→ 텍스트 직접 추출 우선

2. 페이지 이미지뿐임
→ PaddleOCR로 글자와 위치 추출
```

PaddleOCR이 주는 것:

```text
텍스트 내용 + bbox 좌표
```

예:

```json
{
  "text": "의자의 개수와 학생 수를 비교해 봅니다.",
  "bbox": [[132,260],[752,260],[752,296],[132,296]]
}
```

### 4.4 드래그 질문

목표:

```text
학생이 교재의 두 번째 글을 드래그
→ 작은 팝업
→ AI-DOC에 질문
```

AI-DOC으로 넘길 정보:

```json
{
  "bookId": "hs-thinking-basic",
  "page": 4,
  "blockId": "p004_text_02",
  "selectedText": "드래그한 문장",
  "mode": "question"
}
```

1차는 전체 텍스트 레이어까지 완성하지 않아도 된다.

우선순위:

```text
1차: 중요 개념/문항 텍스트만 선택 가능
2차: 전체 페이지 OCR 텍스트 레이어
3차: 문항 ID + 유사문제 템플릿 연결
```

## 5. 유사문제 모델

유사문제는 자유 생성 AI로 풀면 말이 안 되는 문제가 생긴다.  
정답은 **고정 문항 템플릿 기반 유사문제**다.

```text
완전 자유 AI 유사문제 ❌
고정 문항 템플릿 기반 유사문제 ⭕
```

문항 모델 예:

```json
{
  "problemId": "HS_BASIC_004_01",
  "bookId": "hs-thinking-basic",
  "page": 4,
  "unit": "문제푸는 방법 찾기(1)",
  "type": "과부족",
  "difficulty": "원본 동일",
  "variables": ["학생 수", "의자 수", "남는 수"],
  "answerRule": "고정 계산 규칙",
  "diagramRule": "없음 또는 고정 도형 규칙"
}
```

생성 단계:

```text
1단계: 브라우저 JS로 숫자 변형
2단계: 무료 API 또는 가벼운 AI로 문장만 자연스럽게 다듬기
3단계: DOCSSAM 검산 규칙으로 정답/풀이 확인
```

장점:

```text
- 말도 안 되는 문제 생성 감소
- 정답 검산 가능
- 난이도 유지 가능
- 무료 또는 저비용 가능
- 원본 교재를 직접 복제하지 않고 변형 가능
- 지필드만의 유형 DB 축적
```

## 6. Admin 모델

Admin은 viewer와 반드시 세트다.

### 6.1 강의 버튼 관리

버튼 데이터 필드:

```text
buttonName
videoUrl
targetPage
pageAnchor 또는 scrollSlot(선택)
materialUrl
answerUrl
order
enabled
```

`scrollSlot`은 초기에는 필수 핵심이 아니다.  
페이지 이미지 한 장 맞춤 구조에서는 `targetPage`가 우선이다.  
다만 나중에 페이지 안 특정 위치/문항/텍스트 블록으로 이동할 수 있도록 `pageAnchor` 또는 `scrollSlot` 필드를 열어둘 수 있다.

### 6.2 책 미리보기 기반 설정

Admin에서는 책 미리보기를 보면서 버튼의 대상 페이지를 지정할 수 있어야 한다.

초기 권장:

```text
책 미리보기
→ 현재 페이지 선택
→ 버튼 저장
```

확장:

```text
책 미리보기 + 위치 1~11
또는 OCR/text-layer block 선택
→ targetPage + pageAnchor 저장
```

### 6.3 홈 콘텐츠 관리

홈 콘텐츠는 사용자 화면에 덕지덕지 링크를 보여주기 위한 것이 아니라, 마케팅/플랫폼 프리뷰를 관리하기 위한 것이다.

관리 대상:

```text
- 소개 카피
- 히스토리 릴 자료
- 플랫폼 프리뷰 이미지
- 수학/영어/과학 카드 이미지
- 학부모 반응 캡처(개인정보 흐림 처리 필수)
```

GitHub 게시판은 사용자 화면에 노출하지 않는다.

## 7. 단계별 구현 범위

### 7.1 1차 파일럿 필수

```text
- 홈페이지: 시크릿 서재 소개 + 플랫폼 프리뷰 이미지 중심
- viewer-v2: 영상 4 / 교재 6
- 페이지 이미지 기반 eBook
- 상단 버튼: 영상 전환 + 페이지 이동
- 책만 보기 모드
- 한쪽 보기 / 두쪽 보기
- 좌우 얇은 화살표
- 휠 페이지 넘김
- 오른쪽 확대/축소 컨트롤
- 같은 뷰어 안의 인쇄 모드
- admin: 강의 버튼/영상/페이지 관리
```

### 7.2 1차에서 하지 말 것

```text
- AI-DOC을 메인 기능처럼 크게 만들기
- 교재 페이지 위 AI 컴포넌트 얹기
- 문제별 AI 버튼
- 문제 하나만 확대
- 메모/판서/북마크/검색/썸네일을 모두 한 번에 구현
- PDF iframe 중심으로만 의존
- GitHub 게시판 사용자 노출
- Report 3 / gfield-hq 운영본 직접 연결
```

### 7.3 2차 확장

```text
- 검색 인덱스
- OCR/text-layer
- 드래그 질문
- 문항 ID 부여
- 유사문제 템플릿
- 페이지 위 link/imageSlider/youtube 컴포넌트
- 북마크/메모/판서
```

## 9. 최종 총괄 판단

GFIELD-ON은 “개인이 만드니까 이 정도면 됐다” 수준으로 가면 안 된다.  
기능은 작게 시작해도, 핵심 경험은 제대로 보여야 한다.

학부모가 봤을 때:

```text
자료실이 아니라 진짜 지필드 전용 플랫폼이네.
```

학생이 봤을 때:

```text
영상 보면서 바로 교재를 같이 볼 수 있네.
```

원장님이 봤을 때:

```text
이건 앞으로 확장해도 되겠다.
```

1차의 기준은 이것이다.

```text
이 정도면 됐다 ❌
지금 단계에서 보여줄 핵심은 부끄럽지 않게 끝냈다 ⭕
```
