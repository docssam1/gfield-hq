# WORKER_OPERATING_RULES.md

# GFIELD 작업자 공통 운영 원칙
Karpathy Skills + DOCSSAM 운영 원칙

이 문서는 GFIELD 관련 모든 작업자가 작업 전에 반드시 먼저 읽어야 하는 공통 운영 규칙이다.

적용 대상:
- GPT
- Claude / Claude Code
- Gemini
- Codex
- Python / VM / API 작업자
- PDF / eBook 작업자
- 관제센터 작업자

핵심 원칙:
- 먼저 생각하고, 작게 고치고, 실제로 확인한다.
- 추측하지 않는다.
- 운영본을 보호한다.
- G-FIELD 교재와 문제는 DOCSSAM 기준으로 검산한다.

---

## 1. Think Before Coding

작업 전에 먼저 확인한다.

- 작업 목적
- 현재 구조
- 영향 범위
- 수정 대상 파일
- 건드리면 안 되는 파일
- 운영본/실험본 구분
- 검증 방법

모르는 것은 추측하지 않는다.

추측 금지 대상:
- 경로
- 파일명
- 서버명
- 저장소명
- 쪽수
- 문제 번호
- 기능 범위
- 운영본 위치
- GitHub 반영 여부

애매하면 반드시 이렇게 쓴다.

```text
확인 필요
```

---

## 2. Simplicity First

가장 단순한 해결책을 우선한다.

금지:
- 불필요한 새 구조 만들기
- 새 프레임워크 도입
- 과한 리팩토링
- 전체 파일 재작성
- 요청보다 큰 범위 수정
- 겸사겸사 수정

기준:

```text
버튼 하나 수정이면 버튼 하나만 수정한다.
문구 하나 수정이면 문구 하나만 수정한다.
경로 확인이면 경로만 확인한다.
```

---

## 3. Surgical Changes

요청한 범위만 외과수술식으로 수정한다.

건드리지 말 것:
- 관련 없는 파일
- 기존 정상 디자인
- 기존 정상 함수
- 기존 정상 폴더명
- 기존 정상 데이터 구조
- 운영본 파일
- 민감정보 파일

수정 전 반드시 구분한다.

```text
수정할 것:
건드리지 않을 것:
확인 필요한 것:
```

---

## 4. Goal-Driven Execution

“수정했습니다”로 끝내지 않는다.
가능하면 실제 검증 후 보고한다.

검증 예시:
- 실행 확인
- 화면 확인
- 링크 확인
- 저장 확인
- 모바일 확인
- GitHub Pages 확인
- PDF 인쇄 확인
- PDF 오버플로우 확인
- eBook viewer 확인
- manifest 연결 확인
- 영상 버튼 동작 확인

검증하지 못한 것은 반드시 이렇게 쓴다.

```text
검증 못 함
```

금지:
- 확인하지 않고 정상이라고 말하기
- 실행하지 않고 완료라고 말하기
- 링크를 열지 않고 연결됨이라고 말하기

---

## 5. Production Safety

운영본과 실험본을 절대 섞지 않는다.

보호 대상:
- gfield-report 3
- 현재 운영 리포트 폴더
- 실제 학부모 서비스
- 학생 DB
- 카톡 원문
- 실제 Drive 링크
- API Key
- Token
- .env
- 서비스 계정 JSON

실험 원칙:
- 실험은 별도 폴더
- 실험은 별도 브랜치
- 실험은 별도 파일
- 운영본 반영 전 원장님 승인 필수

금지:
- 운영본 직접 수정
- 운영본과 실험본 자동 브릿지
- 운영본 경로 추측
- 민감자료 GitHub 업로드
- 실제 학생/학부모 정보 출력

---

## 6. G-FIELD Content Accuracy

교재, 문제, PDF, 해설, 유사문제 작업은 DOCSSAM 상고모드 기준으로 한다.

반드시 확인:
- 문제 조건
- 그림 조건
- 답
- 단원
- 차시
- 책 하단 실제 쪽수
- 보기 번호
- 표/그래프 수치
- 도형 연결 상태
- PDF 페이지와 책 쪽수 구분

중요:
- PDF 페이지가 아니라 책 하단 실제 쪽수를 기준으로 본다.
- 그림은 실제 문제 조건과 일치해야 한다.
- 도형, 숫자, 선, 화살표, 텍스트가 겹치면 안 된다.
- 문제 박스 밖으로 그림이 넘치면 안 된다.
- 답란, 풀이란, 해설 박스가 페이지 밖으로 밀리면 안 된다.

검산 기준:
- 조건 1회 확인
- 계산 1회 확인
- 그림/레이아웃 1회 확인
- 최종 답 1회 확인

즉 최소 2회 이상 검산한다.

---

## 7. GFIELD eBook / PDF Layout Safety

eBook 또는 PDF 교재 작업 시 다음을 지킨다.

- A4 세로 고정
- `div.page` 단위 페이지 구조
- 페이지 밀림 방지
- 푸터와 페이지 번호 absolute 고정
- 중요한 배경은 SVG 배경 레이어 사용
- 최종 eBook은 page-001.webp 형식 고려
- book-manifest.json 연결 고려

금지:
- 반응형으로 A4 크기 흔들리게 만들기
- PDF iframe 중심으로만 의존하기
- 페이지 안의 요소가 다음 페이지를 밀게 만들기
- 영상 없는 페이지에서 빈 오류 패널 노출

---

## 8. Automation Safety

자동화는 상시 감시가 아니라 단발 호출 구조로 한다.

기본 구조:

```text
수동 투입
→ 텔레그램 단발 호출
→ 1회 확인
→ 대기 목록 보고
→ 원장님 승인
→ 단발 처리
→ 결과 저장
→ 보고 후 종료
```

금지:
- 30초 polling
- 상시 감시 루프
- pc_agent.py --interval 30
- PC↔VM queue/result 자동 연결
- 자동 변환
- 자동 업로드
- 자동 삭제
- 자동 공개
- 반복 상태 보고

---

## 9. Report Format

작업 후 보고는 길게 쓰지 않는다.
반드시 아래 형식으로만 보고한다.

```text
수정한 것:
-

건드리지 않은 것:
-

검증한 것:
-

검증 못 한 것:
-

다음에 해야 할 것:
-
```

긴 로그 금지.
긴 변명 금지.
추측 금지.
필요 없는 설명 금지.

---

## 10. Final Rule

작업자는 항상 아래 문장으로 판단한다.

```text
지금 이 작업이 운영본을 위험하게 만들지 않는가?
지금 이 작업이 요청 범위를 넘지 않는가?
지금 이 작업을 실제로 검증했는가?
```

세 질문 중 하나라도 불확실하면 멈추고 보고한다.

```text
확인 필요. 실행하지 않음.
```
